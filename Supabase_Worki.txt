Supabase Integration - Complete Working Specification
====================================================

1) Purpose
----------
This project uses Supabase as the canonical persistence backend for CSV-shaped datasets via a path->content model in table `app_files`.

2) Core Integration Modules
---------------------------
A. Services/app/supabase_store.py
- Provides `SupabaseFileStore`:
  - read_text(path)
  - write_text(path, content)
  - list_paths(prefix)
  - read_csv(path, columns)
  - write_csv(path, dataframe)
- Includes retry/backoff for transient errors.

B. Services/app/storage.py
- Provides `DataStorage` abstraction used by all UI tabs.
- Routes reads/writes across:
  - Supabase (preferred when configured),
  - sqlite (optional),
  - local mirror fallback.
- Tracks last write health:
  - ok
  - backend
  - path
  - remote_ok
  - local_ok
  - error
- Emits structured storage logs via `log_event`.

C. restore.py
- Imports legacy GitHub CSV datasets into active storage backend.
- Uses public `import_legacy_csv` with idempotency (skip unchanged).

4) Data Model
-------------
Table: app_files
Required columns:
- path (text, unique or primary key)
- content (text)
Recommended:
- updated_at (timestamp, optional)

Storage convention examples:
- Data/TMS_Data/tms_ledger_master.csv
- Data/TMS_Data/tms_holdings.csv
- Data/Logs/activity_log.csv
- Data/Market_Data/Stock_Data/<SYMBOL>.csv

5) Runtime Save Workflow
------------------------
1. UI action calls a storage save API.
2. Storage normalizes DataFrame schema.
3. Storage attempts Supabase upsert with retry/backoff.
4. Storage writes local mirror file.
5. Storage records write health status and logs a `storage_write` event.
6. On failure, storage raises exception so UI can show explicit error (no false success).

6) Runtime Read Workflow
------------------------
1. Try Supabase read first (if active).
2. Fallback to local mirror file.
3. Fallback to empty DataFrame with expected columns.

7) Health & Diagnostics
-----------------------
- Sidebar shows active backend.
- Manage Data tab includes storage health panel.
- App writes structured events to `logs/app_events.jsonl`.

8) Restore/Migration Behavior
-----------------------------
- Restore scans GitHub logical CSV paths and directory CSV files.
- Each file imported via idempotent API:
  - changed files -> imported
  - identical files -> skipped
- Summary shows imported/skipped/failed.

9) Known Limits
---------------
- CSV-blob strategy still has last-write-wins under concurrent edits.
- Large files can increase latency.
- Full overwrite on save is not row-level transactional.

10) Recommended Evolution
-------------------------
- Add optimistic concurrency/version tokens.
- Add CI integration tests against Supabase test environment.
- Migrate high-volume entities to normalized relational tables.
