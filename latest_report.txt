Latest Architecture & Improvement Report
=======================================
Date: 2026-03-02
Repository: Copy-Paste

Executive Summary
-----------------
This revision implemented the major improvement backlog previously identified:
1) explicit storage-write reliability and visibility,
2) retry/backoff for transient Supabase failures,
3) idempotent restore behavior,
4) structured storage event logging,
5) diagnostics surfacing in UI,
6) stronger regression testing for storage paths.

What Was Improved (Implemented)
-------------------------------
1. Reliability Hardening
- Added retry/backoff in Supabase REST operations (`SupabaseFileStore`) for read/list/write operations.
- Write path now raises on persistent failure instead of silently succeeding.
- Storage write status is captured in-memory (`last_write_status`) and exposed via health helpers.

2. Observability and Diagnostics
- Added structured storage event logging via `log_event` for every storage write attempt.
- Added storage health panel in Data Management UI to inspect last write path/backend/error/flags.
- Kept backend indicator in sidebar to reduce deployment confusion.

3. Save-Flow Safety in UI
- Wrapped key save operations in transaction/holding/manual entry flows with try/except and user-facing failure messages.
- Prevents misleading success notifications when persistence fails.

4. Restore Robustness
- Reworked restore logic to be idempotent using public `import_legacy_csv` API.
- Added "skipped unchanged" accounting to avoid duplicate/no-op reimports.
- Removed direct reliance on internal `_save` in restore flow.

5. API Improvements
- Added `import_legacy_csv` public API to storage abstraction.
- Added `get_storage_health` and `last_write_ok` for operational checks.

6. Test Coverage Improvements
- Added storage test for:
  - Supabase usage when backend value is `csv` but Supabase is configured.
  - Idempotent legacy import behavior.

Current Good Sides
------------------
- Centralized storage abstraction with runtime backend introspection.
- Better reliability and failure visibility for transaction persistence.
- More production-friendly restore migration behavior.
- Improved diagnostics and event traces for debugging cloud incidents.
- Backward-safe local mirror still available for offline fallback.

Remaining Bad Sides / Open Risks
--------------------------------
1) Data model is still CSV-blob based per path:
- Large datasets can be slow to overwrite.
- Concurrent writes are still last-write-wins.

2) No live Supabase integration tests in CI:
- Unit tests cover behavior, but not end-to-end auth/network/table policy issues.

3) No conflict/version metadata on writes:
- Parallel sessions can overwrite changes.

4) Diagnostics are local-app oriented:
- No external monitoring/alerting yet.

Future Problems to Watch
------------------------
- Multi-user race conditions on same logical file.
- Growing payload size and write latency.
- Potential schema drift if CSV columns evolve without migration policy.

Priority Next Steps
-------------------
P0
- Add optional optimistic concurrency token (hash/version) in `app_files` rows.
- Add integration test suite with Supabase test project in CI.

P1
- Add write-latency and error counters to admin dashboard.
- Add explicit migration/version metadata per logical dataset.

P2
- Gradually migrate critical entities (ledger, holdings, transactions) from CSV blobs to normalized Supabase tables.

Final Assessment
----------------
Architecture is now significantly more robust than earlier versions and addresses the primary trust issue (false success on failed persistence). The remaining limitations are mostly strategic scalability concerns, not immediate correctness blockers.
